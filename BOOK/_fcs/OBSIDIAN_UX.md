# Obsidian UX & Operator Manual

Obsidian serves as the **human operator interface** for the Fractal Content Studio (FCS). It provides the visual, navigational, and command execution layer over the underlying file-based architecture.

> **Important:** Obsidian is an interface, not the source of truth. The true state resides in the YAML frontmatter, `manifest.md` files, Python scripts, and the Git repository.

## 1. Core Principles of the Obsidian UX

- **Read-Heavy, Command-Driven:** The Founder rarely types prose directly. Instead, they read rendered outputs, navigate dashboards, and trigger commands (via Templater/QuickAdd) to instruct the AI.
- **Data-Driven Views:** All dashboards and lists are dynamically generated using Dataview, querying the YAML frontmatter across the repository.
- **Visual Spatialization:** Graph View and Canvas are used to understand dependencies and conceptual clusters, not just folder structures.
- **Frictionless Action:** Creating an Action Request must take fewer than 3 keystrokes or clicks.

## 2. The Plugin Stack & Configuration

The FCS relies on a specific set of Obsidian community plugins. See [PLUGIN_STACK.md](PLUGIN_STACK.md) for detailed configuration.

1. **Dataview:** Powers all dashboards (Structure, Action, Review, Resources).
2. **Templater:** Powers the generation of new nodes, concepts, resources, and action requests with dynamic YAML injection.
3. **QuickAdd:** Provides the Command Palette interface to rapidly trigger Templater scripts without navigating folders.
4. **Graph View & Local Graph:** Visualizes `parent`, `depends_on`, and `supports` relations.
5. **Canvas:** Used for visual storyboarding and spatial arrangement of modules before committing them to the manifest.

## 3. Operator Workflows

### 3.1 Navigating the Structure
- The Operator starts at the **Structure Dashboard** (`BOOK/views/dashboards/STRUCTURE_DASHBOARD.md`).
- They click into a specific Part or Foundation to view its `index.md`.
- They review the `manifest.md` to see the ordered sequence of children.

### 3.2 Creating a New Node
1. Operator triggers QuickAdd (e.g., `Cmd+Shift+N`).
2. Selects "Create New Module".
3. QuickAdd prompts for the Title and Parent ID.
4. Templater generates the file from `module_template.md` in the correct directory.
5. Operator manually adds the new module to the parent's `manifest.md`.

### 3.3 Issuing an Action Request
1. Operator navigates to a node that needs work (e.g., `F01_01_M01`).
2. Triggers QuickAdd (e.g., `Cmd+Shift+A`).
3. Selects "Create Action Request".
4. Templater generates a new file in `action_requests/` linked to the current node.
5. Operator fills in the prompt and sets `status: READY_FOR_AI`.

### 3.4 Reviewing AI Output
1. Operator checks the **Review Dashboard** (`BOOK/views/dashboards/REVIEW_DASHBOARD.md`).
2. Opens nodes marked `REVIEW_PENDING`.
3. Reads the rendered content (often using the `review` view generated by `render.py`).
4. Updates the status to `REVISION_REQUIRED` (and creates a new Action Request) or `REVISED` / `LOCKED`.

## 4. Visual Language

- **Tags:** Use Obsidian tags sparingly, primarily for ephemeral states not captured in YAML (e.g., `#urgent`, `#idea`).
- **Callouts:** Use Obsidian callouts (`> [!info]`) in templates to guide the AI and the Operator.
- **Properties View:** Keep the Obsidian Properties view enabled to easily toggle statuses and edit YAML without breaking syntax.
