# FCS Dynamic Binder Protocol

**Status:** CANONICAL
**Version:** v0.1
**Phase:** Phase III — FCS Dynamic Binder
**Last Updated:** 2026-06-28
**Author:** Manus (Executive Orchestrator)

---

## 1. Purpose

Define the FCS Binder as a living Scrivener-like navigation system for Obsidian and Git-based manuscript production.

The binder provides:
- a real-time navigable view of the full manuscript structure
- a handoff/export snapshot for ChatGPT, Manus, GitHub review, and archival
- a source-of-truth registry for module identity, status, and position

---

## 2. Core Principle

**`binder_view.md` is a generated snapshot, not source of truth.**

Never edit `binder_view.md` directly to change manuscript structure.
Always edit the source of truth (frontmatter / registry / manifests), then regenerate.

---

## 3. Source of Truth

The authoritative binder data lives in, in order of precedence:

| Source | Location | Role |
|---|---|---|
| Module frontmatter | `BOOK/manuscript/**/*.md` | Per-file metadata (id, title, type, status, parent, order) |
| Central registry | `BOOK/_fcs/registries/book_module_registry.yaml` | Flat list of all modules with paths and statuses |
| Part manifests | `BOOK/manuscript/<part>/manifest.md` | Part-level structure and ordering |
| Module manifests | `BOOK/manuscript/<part>/<module>/manifest.md` | Module-level children ordering |

**Conflict resolution:** frontmatter > registry > manifest.

---

## 4. Dynamic View (Obsidian)

Obsidian Dataview pages read module frontmatter and display live binder views.

- `BOOK/_fcs/binder_views/OBSIDIAN_DYNAMIC_BINDER.md` — full manuscript binder
- `BOOK/_fcs/binder_views/OPENING_DYNAMIC_BINDER.md` — Opening Part only

These pages update automatically when frontmatter changes.
**Requires:** Obsidian Dataview plugin.

---

## 5. Static Export (Generated)

Static Markdown binder views are regenerated by `scripts/generate_binder_view.py` for:

- ChatGPT context packs
- Manus handoff
- GitHub review
- reports
- archival snapshots

Generated files:
- `BOOK/_fcs/binder_views/current_binder_view.md` — full manuscript snapshot
- `BOOK/_fcs/binder_views/opening_binder_view.md` — Opening Part snapshot

**Regenerate after any structural change:**
```bash
python scripts/generate_binder_view.py
```

## 6. Required Module Frontmatter

### For manuscript modules

Every manuscript module should include, at minimum:

```yaml
---
id: OPN-001                          # Legacy FCS id (keep for compatibility)
module_id: OPN-001                   # Binder-canonical ID
title: "Module Title"
type: module                         # module | facet | beat | chapter | part
part: OPENING                        # OPENING | FOUNDATIONS | TRANSITION | IMPLEMENTATION | APPENDICES
parent: OPN                          # Parent module_id or part code
order: 001                           # Sort order within parent
status: STRUCTURE_DRAFT              # See status taxonomy below
fcs_role: manuscript_module          # manuscript_module | manuscript_beat | system_file
scale: null                          # null | INDIVIDUAL | COMMUNITY | CIVILIZATION
foundation: null                     # null | F01 | F02 | ... | F07
facet: null                          # null | F01_01 | etc.
word_count_target: 1500
word_count_current: 0
source_status: STRUCTURE_ONLY        # STRUCTURE_ONLY | PROVISIONAL_MANUS_DRAFT | ROUTED | ACCEPTED
prose_status: NOT_WRITTEN            # NOT_WRITTEN | PROVISIONAL_MANUS_DRAFT | DRAFT_0 | ACCEPTED
routing_status: NOT_ROUTED           # NOT_ROUTED | CHATGPT_REVIEWED | CLAUDE_WRITTEN | FULLY_ROUTED
created_by: null
last_updated_by: null
---
```

### For beats/submodules

```yaml
---
module_id: OPN-001-01
title: "Beat Title"
type: beat
part: OPENING
parent: OPN-001
order: 001.01
status: STRUCTURE_DRAFT
fcs_role: manuscript_beat
---
```

### Compatibility note

Existing modules use the legacy `id` field (e.g., `P01_M01_pathology_of_present`).
The new `module_id` field is additive — do not remove `id`.
The generator script reads both; `module_id` takes precedence if present, else falls back to `id`.

## 7. Status Taxonomy

| Status | Meaning |
|---|---|
| `IDEA` | Concept noted, not yet structured |
| `SCAFFOLDED` | FCS file created, frontmatter only, no prose |
| `STRUCTURE_DRAFT` | Structure proposed, not yet validated |
| `STRUCTURE_REVIEW` | Structure submitted for review |
| `STRUCTURE_VALIDATED` | Structure validated by Founder |
| `READY_FOR_MATERIALIZATION` | Validated structure ready for FCS file creation |
| `PROVISIONAL_MANUS_DRAFT` | Prose written by Manus internally, not routed |
| `READY_FOR_ROUTED_WRITING` | Structure validated, ready for ChatGPT + Claude routing |
| `DRAFT_0` | First prose draft via proper routing (ChatGPT structure + Claude prose) |
| `REVIEW_PENDING` | Draft awaiting review |
| `REVISION_REQUIRED` | Review returned with required changes |
| `ACCEPTED` | Reviewed and accepted |
| `ARCHIVED` | Removed from active manuscript |

**Legacy FCS statuses (still valid for non-opening content):**
`PLANNED`, `DRAFT`, `AI_PROPOSAL_READY`, `REVISED`, `LOCKED`

---

## 8. Binder Update Rule

When a module is added, moved, renamed, split, merged, or archived:

1. Update module file frontmatter (`module_id`, `parent`, `order`, `status`)
2. Update `book_module_registry.yaml` (path, status fields)
3. Run `python scripts/generate_binder_view.py` to regenerate static exports
4. Obsidian Dataview binder updates automatically on next Obsidian reload
5. Commit all changes together

---

## 9. Folder/File Move Rule

Moving a module requires:

- Preserving `module_id` unless intentionally renumbering
- Updating `parent` field
- Updating `order` field
- Updating `path` in `book_module_registry.yaml`
- Updating `parent` in child modules if any
- Running `generate_binder_view.py`
- Committing all affected files together

---

## 10. Opening Part Rule

The Opening Part is currently in `PROVISIONAL_MANUS_DRAFT` status.

**No prose writing until structure is validated.**

Required sequence:
1. Refine structure with Founder + ChatGPT Chief Architect (movements → modules → beats → executive summaries → writing briefs)
2. Founder validates: `STRUCTURE_VALIDATED`
3. Manus materializes validated structure into FCS files: `READY_FOR_MATERIALIZATION`
4. Route writing: ChatGPT API (structure/content) → Claude API (prose) → ChatGPT API (review) → Claude API (revision)
5. Promote to `DRAFT_0` only after full routing
